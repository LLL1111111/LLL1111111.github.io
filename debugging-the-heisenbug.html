Chasing a Heisenbug through three time zones

2026-06-28 · 6 min read · by Your Name

debuggingconcurrency

It started, as these things do, with a graph. A single spike in the error dashboard at 09:14 on a Tuesday, gone by 09:16, with no stack trace worth the name. By the time anyone looked, the request had succeeded on retry. Case closed. Except it happened again the next Tuesday.

This is the story of a bug that took me two weeks to catch, and roughly forty seconds to fix. If you've ever debugged something that refused to exist while you were watching, you know the specific flavor of madness I'm describing.

The symptom

Our service enriched incoming orders with data from a cache that was warmed by a nightly job. Most of the time it was fast and correct. Occasionally — rarely — an order would come back missing a field that should always be present. The logs said the cache returned null for a key that, when I checked manually, clearly existed.

The first rule of intermittent bugs: if you can't reproduce it, you don't understand it yet.

The false leads

I burned three days on theories that were wrong but reasonable:

  • A stale cache entry — but the timestamps didn't line up.
  • A serialization issue — but the payloads were byte-identical.
  • Network flakiness to the cache node — the metrics were clean.

The thing about a good false lead is that it's almost right. Each of these was plausible enough to spend a day on, and each one taught me something true about a system that turned out to be irrelevant.

The actual cause

The warming job and the read path both computed the cache key. The job ran in UTC. The read path, thanks to a config default nobody had touched in years, computed part of the key using the server's local time — and one fleet of servers had drifted onto a different zone after a migration.

For most of the day the two keys matched. But for one narrow window, straddling midnight in one zone, they didn't. Tuesdays were a coincidence of the deploy schedule. The bug wasn't in the code I was staring at; it was in the assumption that a "key" was the same thing on both sides.

// the read path, simplified
const key = `order:${userId}:${localDate()}`;  // 🐛 local

// the warming job
const key = `order:${userId}:${utcDate()}`;    // ✅ utc

What I took away

Three things stuck with me, and I've since taped a version of them above my desk:

  1. Time is never simple. Any time a bug is periodic, suspect the clock before you suspect anything else.
  2. Two systems that "compute the same key" are two systems, not one. Shared logic that isn't literally shared code will drift.
  3. Reproduce before you fix. The forty-second fix only worked because I'd spent the prior hour making the bug happen on demand.

The fix was a one-line change forcing UTC on both sides, plus a test that would have caught it. The real deliverable was the runbook note, so the next person spends two minutes on this instead of two weeks.
