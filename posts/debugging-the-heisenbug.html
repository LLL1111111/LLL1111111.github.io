Chasing a Heisenbug through three time zones

It only ever showed up in production. Never in staging, never on my laptop, never under the debugger — the moment I looked closely, it behaved. A classic Heisenbug: observation itself seemed to fix it.

The symptom was simple: about once every few thousand requests, a background job would write a stale value back over a fresher one. No exception, no log line, just quietly wrong data that someone would notice hours later.

Narrowing it down

The first breakthrough was realizing it correlated with deploy times across regions. Three time zones meant three rollout windows, and the bug clustered right around the moment two instances briefly ran different code versions against the same queue.

// simplified reproduction
async function updateRecord(id, patch) {
  const current = await db.get(id);
  const merged = merge(current, patch);
  await db.put(id, merged); // read-modify-write, no lock
}

Two workers, running slightly different merge logic during a rolling deploy, could both read the same record, merge independently, and whichever write landed last would win — regardless of which patch was actually newer.

The fix

The real fix wasn't clever: add an optimistic-concurrency check on write, and make the merge idempotent regardless of ordering. The interesting part was how long it took to trust a fix for a bug that reproduced maybe once a day.

If a bug only shows up under specific timing, assume timing is the cause — not the excuse for why you can't find it.

Three weeks later, the counter for "stale write detected" has stayed at zero. That's the only proof I trust for something like this: not that it stopped happening, but that we now know exactly why it can't.
